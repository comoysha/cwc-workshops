把一个 402 行的提示词，
拆成一支 Agent 团队。

StockPilot 是一个中型户外装备零售商的库存管理 Agent。 它管 250 个 SKU、3 个仓库、12 家供应商、90 天销售历史。 v1（Before）把所有业务规则都写进 system prompt，配 12 个工具和 3 个硬编码 subagent。 v2（After）把同样的知识改写成 5 个按需加载的 skill、几个可执行脚本和一个 callable forecaster。 本页给两类读者看： 供应链从业者看的是业务规则去了哪里； IT/Agent 开发者看的是这套架构是怎么被组装出来的。

①这是个什么生意

在写任何 agent 之前，先看清它要服务的业务对象。这不是"几个 SKU 的玩具"—— 它的规模足够让"该不该再下一单"这种小决策每天发生几十次，而每次决策的错都直接掉收入或压库存。

250
SKU（在售商品）

从登山杖到睡袋到头灯，每个有自己的补货点和季节性。

3
仓库

WH-EAST（Carlisle, PA）/ WH-WEST（Reno, NV）/ WH-CENTRAL（Kansas City, MO）。同一 SKU 在三处的库存独立计量。

12
供应商

价格、交付周期、可靠性、MOQ 各不同。还有目录之外的"潜规则"（节假日歇业、品类短发、阶梯折扣）。

90 天
销售历史

用来算速度、识别趋势、对比促销同期。本身大约 6.7 万行。

Agent 要负责的 7 件事

这 7 张卡来自 workshop 首页截图，是 v1 实际承担的业务面。

① 监控与告警

盯库存水平，把跌破补货点的 SKU 浮出水面。

业务含义

当某 SKU 在某仓库的 on_hand < reorder_point，触发评估。 紧急程度按 days_of_cover = on_hand / 日均销量 排序； 覆盖天数 < 供应商交付周期 = 在 PO 到货前必断货。

② 需求预测

尤其在促销和季节切换时。

业务含义

普通日子用 14 天滚动均值即可；遇促销、季节、长 horizon，需要看全 90 天历史寻找类比。 预测必须给出置信度——下游凭它决定"自动下单"还是"升级人工"。

③ 选供应商

价格 vs 交付周期 vs 可靠性。

业务含义

分数 = 0.5×（1−归一化价格）+ 0.3×（1−归一化交付周期）+ 0.2×可靠性。 但目录之外的规则（如 SUP-04 满 250 件 3% 折扣、SUP-02 年末歇业）要手动覆盖。

④ 下单

创建 PO、更新 ERP。

业务含义

补货量 = 日均销量 × 30 + 安全库存 − 当前在手。 向上取整到供应商 MOQ。低于 MOQ 的单会被拒。

⑤ 通知运营

Slack 告警、供应商邮件。

业务含义

例行补货发 #ops-inventory 频道；紧急或大额走升级矩阵（采购主管 / 财务）。 巡检时发一条汇总，而不是每个 SKU 一条。

⑥ 周报

每周一为每个仓库出一份。

业务含义

含本仓最关注项、超过交付周期的未关闭 PO、连续 5 个工作日以上低于补货点的 SKU。 领导看的是趋势，不是事件。

⑦ 监测外部中断

港口延误、物流新闻。

业务含义

会影响交付周期判断——尤其依赖进口的 SUP-07。 v2 直接把这个能力拿掉了：它本属于另一个独立的"市场情报" agent。

数据底盘：6 个文件就是真相

在沙箱里以 CSV / JSONL 形式存在。所有 agent 决策最终都要落到这几张表。

②供应链术语对照（给 IT 读者）

如果你来自 IT、不熟悉供应链，先把这几个词理解清楚，后面的架构图就好懂了。

③三档决策框架：tool → skill → subagent

这是这场 workshop 的核心心智模型。当你想给 agent 加一个能力时，先想清楚它属于哪一档—— 越往右越强，但越贵。Before agent 的问题不是"哪档不对"，而是把好几样东西放错了档位。

最便宜

Tool Call

一个无状态函数。Agent 用一次。

  • 输入参数化、输出固定结构
  • 确定性强、可单测
  • 主成本：每次的 token 进上下文

典型例： get_stock_level(sku, warehouse) 这种"查一个值"。

⚠ 气味：返回 > 2k tokens？换成代码执行。

按需加载

Skill

一段"什么时候触发 + 怎么做"的说明，可附带可执行脚本。

  • 不在常驻 system prompt 里付 token
  • Description 命中关键词才加载
  • 把策略从 prompt 抽离

典型例： 补货公式、供应商打分权重、告警模板。

⚠ 气味：正在 prompt 里写"always do X before Y"？改 skill。

最贵

Subagent

独立的 agent，拥有自己的 context window 和目标。

  • 能隔离大数据/长推理
  • 返回结构化结果
  • 主成本：模型一整次完整启动

典型例： Forecaster 要看完整 90 天历史。

⚠ 气味：subagent 输出是一个数字？它不该是 subagent。

④Before 架构（v1，2025 年的样子）

运行在原生 Messages API 上的手写 agent loop。一个 402 行的 system prompt 把所有业务规则塞进去， 12 个工具直接读 CSV，3 个 subagent 用散文回话再由主 agent 解析。每一处单看都合理，问题在于聚合。

12 个工具（一段时间内每月新增 1–2 个）

    3 个硬编码 subagent

      实现见 agents/before/subagents.py。每个都是独立的 Messages API 调用，返回散文，主 agent 自己从文本里抽数字。

      402 行 prompt 在写些什么？

      把整份 prompt 按章节拆开看，知识密度极高但常驻 context——每一轮模型调用都要付一次 token。

      为什么 v1 跑 F1 任务要 102 次工具调用？

      F1 = 每日低库存巡检。看 agent 在 v1 架构下被迫怎么做：

      ⑤After 架构（v2，部署到 Claude Managed Agents）

      短 prompt + agent_toolset（Bash、文件读写、Task）+ 5 个按需加载的 skill + 1 个 callable forecaster。这套 setup 把"知识"和"算力"重新分配：常驻 context 只剩 15 行， 400 行的 skill 内容只在被用到时进 context。

      5 个 skill（按需触发，平时不进 context）

        1 个 callable forecaster

        一个独立的子 agent，仅在需要看完整 90 天历史时被调用。 主 agent 只发出 SKU、产品标志、horizon 三个参数； forecaster 在自己的 context 里跑 Bash 算预测，只把 4 字段 JSON 返回。

        ⑧四条贯穿全程的设计原则
        
    
        如果你只能从这套 workshop 带走四句话，是这四句。
        
    
        
      
        
        
        1
        
        
        Compute-over-context
        
        
        当你需要遍历/聚合数据时，让代码在沙箱里跑，只把结论放进 context。
        Before agent 把 ~400 行 list_low_stock CSV 全塞回上下文；After agent 一个脚本算完只输出 20 行 JSON。
        
        
        在 forecasting/SKILL.md 体现为：batch_days_of_cover.py 替代 100+ 工具调用。
        
      
        
      
        
        
        2
        
        
        Typed contract
        
        
        当一个 subagent 的输出会被另一个 agent 消费，必须是结构化 JSON，而不是散文。
        每个字段都要有下游消费者。
        
        
        F2 任务 v1 失败的根因：subagent 散文里说"促销可能 3 倍"，主 agent 落到 PO 还是基线均值。
        v2 的 forecaster 返回 {forecast_qty, confidence, method, flags}。
        
      
        
      
        
        
        3
        
        
        Policy-as-skill
        
        
        所有"如果 X 就 Y"的业务规则放进 skill 文件，按需加载而不是常驻 prompt。
        Skill description 列触发关键词，agent 在每轮决定是否拉入。
        
        
        补货公式、供应商打分权重、Slack 模板、周报结构——全部从 prompt 搬到 5 个 SKILL.md。
        
      
        
      
        
        
        4
        
        
        Scope discipline
        
        
        Subagent 只在真的需要独立 context时使用——比如要看完整 90 天历史。
        如果输出是一个数字、或就是个排序，那不该是 subagent。
        
        
        v1 有 3 个 subagent（forecaster / procurement / writing）；
        v2 只留 1 个（forecaster），其它两个被打回了"代码执行"和"模板填充"。

        ⑩给你的 takeaway
        

    
        
      
        
        
        如果你是供应链/运营
        
        
          
          
        • 真正影响产出的不是"agent 用了哪个模型"，而是业务规则被如何表达——是塞在 prompt 里，还是组成一组可单测的小文件？
          • 
          
        • "自动下单 vs 升级人工"的边界要被写成明确阈值（这里是 confidence < 0.6），而不是"凭感觉"。这是审计可解释性的根。
          • 
          
        • 常见陷阱：促销月仍用滚动均值——你的 agent 也会犯。把促销路径和常规路径显式分流（路径 A / 路径 B）。
          • 
          
        • 评估 ROI 不是看模型分数，而是看具体业务事件能否被准时捕获——比如"top-100 SKU 在断货前 24h 是否被告警"。这才是 F1 任务的灵魂。
          • 
        
        
      
        
      
        
        
        如果你是IT/Agent 开发
        
        
          
          
        • 不要把所有能力都做成 tool call。看到工具返回 > 2k tokens、或 prompt 里写"always do X before Y"，就是该拆 skill / 代码执行的信号。
          • 
          
        • Subagent 之间的接口强类型 JSON，不要让主 agent 解析散文——那是 v1 失败的根因。
          • 
          
        • 降级路径设计：当某能力不可用（如 callable_agents 在某些环境没启用），返回一个刻意保守的占位结果（confidence=0.55），让下游阈值自动接住。skill 之间不要硬连，通过阈值耦合。
          • 
          
        • 评估闭环比架构本身更值钱。这个 repo 有 12 个 eval 任务，每个对一个具体业务事件；你的 agent 也应该有。
